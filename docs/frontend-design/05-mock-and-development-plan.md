# Mock 与开发计划

## Mock 策略

第一版前端应先使用 MSW mock 后端接口。

目的：

- 后端未完成时也能开发页面。
- 尽早验证状态、按钮、角色差异和流程。
- 把前端需要的 DTO 提前暴露给后端设计。

## Mock 数据范围

第一批 mock 数据：

- 当前用户
- 一个谷团
- 一个进行中的拼团
- 拼团主图和发起人
- 三个拼单商品
- 当前用户的一条拼单记录
- 当前用户的费用和付款凭证
- 商品图鉴数据和角色标签
- 收款方式
- 可排发商品
- 排发申请
- 国际转运批次
- 转单申请
- 异常记录
- 审计日志
- 管理员用户
- 拼单维护人用户
- 囤货人用户

建议准备角色切换：

- 普通成员
- 拼单维护人
- 囤货人
- 管理员

说明：

- 角色切换只用于本地开发和测试，不一定是正式功能。
- 角色切换能快速验证同一页面的按钮差异。

## 第一阶段开发顺序

### 第 1 步：项目初始化

内容：

- Vite + React + TypeScript
- Tailwind CSS
- Router
- Query Client
- MSW
- 基础 AppShell

完成标准：

- `/login` 可以进入
- 登录后进入 `/app/groups`
- 底部 Tab 可切换

### 第 2 步：登录和启动数据

内容：

- 登录页
- 注册页占位
- `/api/app/bootstrap` mock
- 当前用户状态

完成标准：

- mock 登录后能拿到用户、角色、能力
- 页面能根据角色显示不同入口

### 第 3 步：谷团列表与详情

内容：

- 谷团列表页
- 谷团详情页
- 团信息卡片
- 管理台入口条
- 拼团列表
- 状态筛选

完成标准：

- 手机端接近轻应用体验
- PC 端能正常展开
- 管理员看到编辑和管理台入口
- 普通成员只看到成员能力

### 第 4 步：拼团详情

内容：

- 拼团详情页
- 拼单商品列表
- 我的记录摘要
- 认领按钮
- 管理入口
- 上传付款凭证弹窗
- 付款确认和驳回按钮
- 上传下单截图弹窗

完成标准：

- 普通成员能 mock 认领商品
- 认领后生成 `未肾` 状态记录
- 维护人能看到编辑和管理操作入口
- 成员能从拼团详情打开付款凭证弹窗
- 维护人能在列表中确认或驳回付款
- 维护人能上传下单截图

### 第 5 步：管理台占位

内容：

- 管理台模块入口页
- 根据角色展示可用模块

完成标准：

- 管理员、维护人、囤货人看到不同模块
- 普通成员无法进入管理台

### 第 6 步：新建和编辑拼团

内容：

- 新建拼团
- 编辑拼团
- 商品、库存和内定库存编辑
- 收款方式选择和快速新建
- 从商品图鉴选择商品

完成标准：

- 新建和编辑共用一套表单
- 可保存草稿或发布
- 可在页面内维护商品和库存

## 第二阶段开发顺序

第二阶段再进入流程测试页：

- 商品图鉴列表、筛选和商品详情
- 我的拼单和我的记录
- 我的费用
- 收款方式
- 可排发商品
- 创建排发申请
- 拼单记录列表
- 国际批次
- 囤货工作台
- 排发申请处理
- 转单弹窗和转单审核
- 异常处理
- 用户与角色管理
- 审计日志
- 地址簿

## 页面验收清单

每个页面需要检查：

- 手机宽度下无横向滚动
- 底部 Tab 不遮挡内容
- PC 宽度下内容不显得过窄
- 角色不同，按钮显示正确
- Loading 状态可见
- 空状态可见
- 错误状态可重试
- 长标题和长昵称不会撑坏布局

## 与后端联调策略

建议：

- 前端先按 [04-frontend-api-needs.md](04-frontend-api-needs.md) 写 API client。
- 后端完成接口后逐个替换 MSW handler。
- 每替换一个接口，保留 mock 作为测试数据来源。
- 前端不要直接依赖后端数据库字段名，统一通过 DTO 映射。
