# 已确认与待确认问题

这些问题会影响后续数据库和界面设计，建议在编码前确认。

## 已确认决策

- 成员参团方式：第一版允许普通成员直接在系统里选择商品参团。
- 拼单可见范围：目前只有一个群组，先按单群实现；真实模型应支持“群包含用户”，后续多群再拆出群成员关系。
- 首款未付是否允许进入下单：允许维护人先下单，但系统需要明确标记未付款状态，方便后续追款。
- 税费均摊对象：默认在本次国际运输涉及的全部订单成员中均摊，算法暂不急，但必须支持手动编辑。
- 转单确认规则：转出方和转入方一般先线下确认，再提交系统申请，由维护人或管理员审核通过后生效。
- 图片存储：付款截图、收款二维码、订单截图第一版使用本地 MinIO 服务存储，接口层仍保留 image_url / 对象 URL 抽象。

## 待继续确认

### 1. 费用模型是否支持部分付款

一笔费用是否允许拆成多次付款确认？

建议：

费用模型已经预留 PaymentProofAllocation，可以支持部分付款；第一版界面可以先限制为一次付清，后端结构保留扩展能力。

### 2. 一张付款凭证是否可以抵扣多笔费用

成员是否可能一次转账同时支付首款、国际费或国内运费？

建议：

后端保留一张凭证关联多笔费用的能力；第一版产品流程可以先按单笔费用上传，等对账需求明确后再开放批量核销。

### 3. 退款是否需要上传凭证并审核

退款是否和付款一样要求截图、金额、时间和审核记录？

建议：

退款也作为 Charge 管理，并允许上传退款凭证；金额方向要在费用类型和付款方向里表达清楚，避免负数金额语义混乱。

### 4. 地址管理

是否需要保存成员常用地址？

建议：

第一版可以保存排发申请中的地址快照。常用地址簿可以后续增加。

### 5. 是否需要导出表格

群内管理可能仍需要 Excel 或 CSV。

建议：

保留导出接口，方便维护人对账和群内公示。

### 6. 已切后的剩余库存展示

“已切”状态下，拼主可能已经下单但仍在拼出自己不想要的部分。系统需要确认普通成员看到的是总库存、剩余可拼库存，还是只看到可拼部分。

建议：

成员端只展示可拼部分；维护人端展示总库存、已认领、内定和剩余可拼。

### 7. 调价影响范围

已生成应收或已付款后调价，是否影响所有已参团成员，还是只影响调价后的新参团成员？

建议：

调价申请中明确影响范围：仅新认领、全部未付款成员、全部成员。审核通过后再生成对应补款或退款记录。
